# flsingbox 核心集成架构开发设计需求
**参考项目：** `chen08209/FlClash`  
**目标项目：** `flsingbox`  
**目标：** 参考 FlClash 的核心管理模式，设计并实现一套适用于 `sing-box` 的跨平台核心集成方案。

---

# 1. 项目背景

FlClash 的架构显示，它并不是简单把代理逻辑直接写在 Flutter 层，也不是要求用户手动额外安装核心，而是采用了：

- Flutter 作为 GUI 客户端
- 核心随客户端一起构建/打包/发布
- 桌面端使用独立核心进程
- Android 使用原生库方式集成
- GUI 与核心之间通过本地通信机制协作
- 桌面端针对权限与系统能力做额外适配

`flsingbox` 需要参考这种思路，但底层核心从 ClashMeta 替换为 `sing-box`。

---

# 2. 设计目标

flsingbox 的核心集成方案必须实现以下目标：

1. **sing-box 核心随客户端一同分发**
   - 不能要求用户单独下载核心
   - 核心必须由打包系统集成到发布产物中

2. **桌面端采用独立核心进程模式**
   - GUI 与核心分进程运行
   - 便于升级、崩溃隔离、日志采集和权限管理

3. **移动端采用适合移动平台的集成方式**
   - Android 优先考虑原生集成/动态库方案
   - 同时预留独立二进制或混合集成方案的可行性分析

4. **GUI 与核心之间存在稳定的控制通道**
   - 配置下发
   - 启停控制
   - 状态回传
   - 日志采集
   - 错误回传

5. **权限与系统能力可控**
   - Windows 管理员服务/辅助进程
   - macOS / Linux 核心授权与执行权限
   - Android VPN 能力桥接

6. **可构建、可打包、可发布**
   - 核心构建或下载方案清晰
   - 构建产物按平台集成到 App 中
   - 运行时可定位核心路径

---

# 3. 总体架构要求

---

## 3.1 总体分层

### A. GUI 层
技术建议：
- Flutter / Dart

职责：
- 展示界面
- 接收用户操作
- 管理配置
- 调用核心控制接口
- 显示运行状态、日志、错误

---

### B. Core Manager 层
职责：
- 获取核心路径
- 启动/停止/重启核心
- 检查核心是否存活
- 管理通信连接
- 管理日志与事件
- 管理核心授权状态

---

### C. Runtime Config 层
职责：
- 将 GUI 的节点、路由、DNS、策略组等数据模型生成 sing-box 配置
- 输出运行时配置文件
- 在启动前进行配置合法性校验

---

### D. IPC / Bridge 层
职责：
- 建立 GUI 与核心之间的通信机制
- 在桌面端使用本地 socket / pipe / HTTP API
- 在 Android 端通过 MethodChannel + Native Bridge 连接

---

### E. Platform Integration 层
职责：
- Windows：helper service / 权限提升 / 系统代理
- macOS：权限、文件授权、Network/TUN 适配
- Linux：执行权限、sudo 授权、TUN 支持
- Android：VPN Service、前台服务、native bridge

---

# 4. 核心分发模式要求

---

## 4.1 总原则
`sing-box` 核心必须随 flsingbox 客户端一起分发。

不允许默认依赖以下模式：
- 用户自行下载核心后填写路径
- 首次启动在线下载核心后再运行
- 仅文档提示用户手动准备核心

这些模式可作为高级开发者选项存在，但**不能作为默认交付方案**。

---

## 4.2 平台分发模式

### 桌面端（Windows/macOS/Linux）
必须采用：

> **客户端发行包内包含 sing-box 独立可执行文件**

要求：
- 和客户端一起打包
- 安装后位于应用资源目录或运行目录
- GUI 能通过统一路径解析逻辑找到它
- GUI 直接启动该可执行文件

---

### Android
第一阶段允许两种方案中选一种：

#### 方案 A：native library 集成
- 将 sing-box 打包为 `.so`
- 通过 Android 原生层调用
- 适合更深入系统集成

#### 方案 B：打包为可执行资源并在运行时释放
- App 内携带核心
- 首次运行释放到内部目录
- 设置可执行权限后启动

要求：
- 开发实现前必须先评估 sing-box 在 Android 上的最佳集成模式
- 若选方案 B，必须说明安全性、兼容性、可维护性影响

推荐：
- **先做桌面端成熟方案**
- **Android 单独做集成预研后再定**

---

# 5. 核心来源要求

---

## 5.1 核心获取方式
必须支持以下至少一种：

### 方式 A：构建时下载指定版本 sing-box 二进制
优点：
- 实现快
- 避免项目仓库承担完整核心源码构建复杂度

### 方式 B：从 sing-box 源码构建
优点：
- 可定制
- 版本完全可控

建议：
- 第一阶段优先实现 **方式 A**
- 第二阶段支持 **方式 B**

---

## 5.2 核心版本管理
必须有统一变量管理：

- `APP_VERSION`
- `CORE_VERSION`

要求：
- 核心版本必须可单独切换
- App About 页面应展示当前核心版本
- 构建脚本根据核心版本决定下载/构建内容
- Release Notes 必须注明使用的 sing-box 版本

---

## 5.3 核心校验
建议但强烈推荐：
- 对核心文件做 SHA256 校验
- 构建时校验下载内容完整性
- 运行前可选做版本/校验检查

---

# 6. 核心路径与资源布局要求

---

## 6.1 桌面端核心路径
必须提供统一路径解析层：

### 建议目录
#### Windows
- `appDir/sing-box.exe`
- 或 `appDir/data/core/sing-box.exe`

#### macOS
- `.app/Contents/Resources/sing-box`
- 或 `.app/Contents/MacOS/resources/sing-box`

#### Linux
- `AppDir/usr/bin/sing-box`
- 或应用目录下 `resources/sing-box`

要求：
- 不同平台的路径差异封装在 `AppPath/CorePathResolver` 中
- Flutter 上层不直接拼路径字符串

---

## 6.2 Android 核心路径
如采用释放方案：
- 释放到 App 内部私有目录
- 例如 `filesDir/core/sing-box`
- 首次启动检查是否存在，不存在则释放
- 释放后设置执行权限

如采用 `.so` 方案：
- 由 Android 原生层负责装载
- Flutter 不感知文件路径，只感知桥接接口

---

## 6.3 Helper / 辅助进程路径
若桌面端需要辅助服务，也必须统一管理：
- Windows helper service
- 其他平台潜在辅助组件

---

# 7. 桌面端核心运行模式要求

---

## 7.1 独立进程模式
桌面端必须采用：

> **Flutter GUI 进程 + sing-box 独立核心进程**

要求：
- GUI 不直接把核心逻辑嵌入主进程
- GUI 通过 `Process.start` 或平台等效方式启动 sing-box
- 核心崩溃不应直接导致 GUI 崩溃

---

## 7.2 启动流程
启动流程必须至少包含：

1. GUI 检查核心文件是否存在
2. GUI 检查核心权限是否满足运行要求
3. GUI 生成运行时配置文件
4. GUI 建立 IPC 监听
5. GUI 启动 sing-box 进程
6. 核心连接到 GUI 或 GUI 连接到核心控制接口
7. GUI 标记运行状态为已启动

---

## 7.3 停止流程
停止流程至少包含：

1. GUI 请求核心优雅退出
2. 等待短暂超时
3. 若未退出则强制 kill
4. 清理通信连接
5. 更新 UI 状态
6. 恢复必要的系统设置（如 DNS / 系统代理）

---

## 7.4 重启流程
必须支持：
- 配置变更后自动重启核心
- 手动重启
- 核心崩溃后重启（可选自动）

---

# 8. GUI 与核心通信方式要求

---

## 8.1 通信原则
GUI 与核心必须有明确可维护的通信机制。

优先选择以下之一：

### 方案 A：本地 socket
适合参考 FlClash 桌面端模式  
优点：
- 实现清晰
- 跨平台可行
- 可做双向通信

### 方案 B：本地 HTTP API
如果 sing-box 提供适合的 API，可利用本地控制接口  
优点：
- 简单直观
- 易调试

### 方案 C：stdin/stdout + 日志流 + 配置文件
仅作为极简 fallback，不建议作为完整方案

建议：
- **桌面端优先本地 socket / 本地控制接口**
- **Android 通过原生桥接**

---

## 8.2 通信能力要求
必须至少支持：
- 启动确认
- 停止确认
- 配置加载结果
- 当前运行状态回传
- 错误事件
- 核心崩溃事件
- 实时日志输出

---

## 8.3 消息结构
如果使用自定义 socket 协议，建议采用 JSON line 模式：

```json
{"type":"event","event":"started"}
{"type":"event","event":"log","level":"info","message":"..."}
{"type":"event","event":"crash","message":"..."}
{"type":"request","id":"xxx","method":"reload"}
{"type":"response","id":"xxx","success":true}
```

要求：
- 事件与请求响应分离
- 支持 request id
- 支持超时处理

---

# 9. 日志管理要求

---

## 9.1 日志来源
必须能采集：
- 核心 stdout
- 核心 stderr
- GUI 自身日志
- 平台桥接日志（可选）

---

## 9.2 日志能力
必须支持：
- 实时显示
- 本地缓存
- 按级别过滤
- 导出日志
- 启动失败日志可见

---

## 9.3 核心崩溃日志
若核心崩溃：
- UI 应立即收到事件
- 日志页应能看到最后日志
- 可给出“重启核心”按钮

---

# 10. 配置文件管理要求

---

## 10.1 配置生成职责
核心层不直接感知 Flutter UI 结构。  
配置生成必须由 `RuntimeConfigBuilder` 完成。

输入：
- 节点
- 分组
- 路由规则
- DNS 配置
- TUN 配置
- 实验项

输出：
- sing-box JSON 配置文件

---

## 10.2 配置文件路径
必须使用统一运行目录，例如：
- `appSupportDir/runtime/config.json`

要求：
- 每次启动前更新
- 保留最近一次成功配置
- 支持预览当前运行配置

---

## 10.3 启动前校验
启动核心前必须执行：
- 配置文件存在性检查
- JSON 合法性检查
- 必填字段校验
- 节点与 detour 链路合法性校验

---

# 11. 权限与授权要求

---

## 11.1 Windows
参考 FlClash，需要设计 Windows 提权能力。

要求：
- 若普通权限无法满足核心运行，需要提供 helper service / elevated launcher
- 支持检测 helper/service 是否存在、是否运行
- 支持安装和启动 helper service
- 权限相关错误应可见

### 建议实现
- 单独 helper 可执行文件
- 负责管理员权限下启动/停止核心
- 管理系统代理/TUN 等高权限行为

---

## 11.2 macOS
要求：
- 检查核心文件权限
- 如需高权限运行，提供授权流程
- 如涉及 TUN / Network Extension，必须说明后续签名和系统权限要求

第一阶段允许：
- 先实现开发版授权和本地运行流程
- 高级系统能力后续增强

---

## 11.3 Linux
要求：
- 检查核心文件是否可执行
- 必要时执行 chmod
- 如需 TUN 等高权限，支持 sudo 授权流程
- 提示用户安装必要依赖

---

## 11.4 Android
要求：
- VPN Service 接入明确
- 前台服务生命周期明确
- 原生层与 Flutter 的交互接口明确
- 权限申请与错误反馈清晰

---

# 12. Android 集成特别要求

---

## 12.1 Android 核心运行架构
必须完成预研并选择一种：

### 方案 A：JNI / `.so` 集成
- sing-box 编译为 Android so
- 原生层调用
- Flutter 通过 MethodChannel 调度

### 方案 B：独立可执行释放运行
- App 资源内带核心
- 运行时释放和 chmod
- 使用 Process / native 启动机制

---

## 12.2 Android 桥接接口
至少提供以下 Flutter 调用接口：
- startCore(configPath / runtimeOptions)
- stopCore()
- restartCore()
- getCoreStatus()
- streamLogs()
- updateVpnState()

---

## 12.3 Android 生命周期
必须明确：
- App 前后台切换时核心行为
- VPN 服务生命周期
- 用户手动关闭时的处理
- 核心异常退出时的回调

---

# 13. 构建与打包要求

---

## 13.1 构建系统
需要参考 FlClash 提供统一构建入口，例如：

```bash
dart setup.dart windows --arch x64
dart setup.dart linux --arch x64
dart setup.dart macos --arch arm64
dart setup.dart android
```

该脚本负责：
1. 准备核心
2. 注入核心到平台工程
3. 调用 Flutter build
4. 整理发布产物

---

## 13.2 平台打包逻辑
### 桌面端
- 把 sing-box 可执行文件拷入发行目录
- 把 helper/service 一并拷入（若有）
- 确保最终安装包中包含核心

### Android
- 把 `.so` 或资源核心打入 apk/aab
- 运行时可访问

---

## 13.3 运行时版本展示
应用内必须显示：
- flsingbox app version
- sing-box core version

建议显示位置：
- About 页面
- 日志页
- 错误诊断页

---

# 14. 失败恢复与健壮性要求

---

## 14.1 核心缺失
若核心文件不存在：
- 提示“客户端资源不完整”
- 不得无提示崩溃
- 提供诊断信息

## 14.2 核心启动失败
若启动失败：
- 显示失败原因
- 显示 stderr/错误日志
- 不污染上次成功状态

## 14.3 核心崩溃
若核心运行中崩溃：
- UI 应收到 crash 事件
- 提供重启入口
- 保留崩溃前最后日志

## 14.4 配置错误
若配置无效：
- 启动前即阻止
- 指出字段错误或映射失败
- 不得直接黑盒失败
